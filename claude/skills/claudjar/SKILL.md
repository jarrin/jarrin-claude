# claudjar

`claudjar` is the CLI that manages this machine's Claude Code config and, per repo, its
git worktrees and their runtime stacks. When a request falls in its territory, drive it
through `claudjar` — not through the underlying `git` / `docker` commands.

**Why this matters:** the raw commands skip everything claudjar layers on top. A bare
`git worktree add` produces a worktree with no copied `.env`, no `PROJECT_PORT`, no
stamped identity, and no setup run — so its stack cannot start and the `todo` /
`staged-planning` skills cannot scope work to it. A bare `docker compose up` binds the
wrong port. Prefer the wrapper; fall back to raw git only for things claudjar has no
command for.

## Does this repo use claudjar?

It applies when the repo has a `.claude/.jarrin.yml`. Check with `claudjar info`, which
prints the resolved config: rules, command table, `project:` stack, backlog methods,
worktree settings, and available skills. If the file is missing, the repo has not opted
in — `claudjar init` scaffolds it.

## The command surface

`references/commands.md` holds the **full** help for every command, generated by
`claudjar help --full`. Read it when you need exact flags; the map below is the routing
table.

| The user wants…                          | Command                                     |
| ---------------------------------------- | ------------------------------------------- |
| A new worktree to work in                | `claudjar worktree create <name>`           |
| To switch to another worktree            | `claudjar goto <name>` (`main` → original)  |
| To fold a worktree's work back           | `claudjar worktree merge <name>`            |
| …and be done with that worktree          | `claudjar worktree merge <name> --remove`   |
| To delete a worktree without merging     | `claudjar worktree remove <name>`           |
| To see the repo's worktrees              | `claudjar worktree list`                    |
| To start / stop the project's services   | `claudjar start` / `claudjar stop`          |
| To cut / tag a release                   | `claudjar release [--bump major\|minor]`    |
| To see this repo's resolved config       | `claudjar info`                             |
| To activate a repo / change its rules    | `claudjar init`                             |
| To set up this machine                   | `claudjar install`                          |

## Things worth knowing before you run one

**Worktree names compose.** `create x` run inside worktree `dev` yields `dev-x`, as a
flat sibling — never nested. Running it in the main checkout gives plain `x`.

**Creating does not start anything.** `create` assigns the worktree a `PROJECT_PORT` and
runs the configured `setup:` commands, but the stack stays down. Run `claudjar start`
inside the new worktree (or tell the user to), and note the statusline dot: green `●`
means the stack is listening, dim `○` means it is not.

**The stack is fully manual.** No session hook starts or stops it. The only automatic
teardown is removing a worktree.

**Merging keeps the worktree.** `merge` folds the branch in and leaves the worktree and
branch alone, because merging up is not the same as being finished. Add `--remove` to
clean up in the same step. On a conflict nothing is removed and an interactive `claude`
resolves it.

**Removing is safe by default.** `remove` stops the stack, deletes the directory, then
deletes the branch with `git branch -d` — which refuses on unmerged work, so the branch
survives and is reported rather than discarded. Run it from a *different* worktree.
`--no-teardown` skips stopping the stack.

**`goto` launches, it does not `cd`.** A process cannot move its parent shell, so `goto`
starts a fresh `claude` in the target worktree. That is deliberate: launching is what
makes the target's SessionStart hook run, giving the new session its stamped identity
and port.

**`release` refuses more than it does.** It runs only on `main` with a clean tree, and
never pushes. If it refuses, relay the reason — do not work around it by tagging or
committing by hand. The version lives in `project.dist.version`; never hand-edit a version
in `package.json`, `pyproject.toml`, or `.env` in a repo that has a `sync:` list, because
`release` owns those and will overwrite them.

## Do not run these

`claudjar api …` (`session-start`, `statusline`) is the internal surface Claude Code
invokes itself, registered in `claude/settings.json` as `$HOME/.local/bin/claudjar api …`.
Those read a JSON payload on stdin and write a format their caller parses. Running them by
hand does nothing useful.

## Editing the config

Config lives in the repo, never in `~/.claude/` (those are symlinks to the jarrin-claude
repo). `.claude/.jarrin.yml` is committed and shared; `.claude/.jarrin.local.yml` is
gitignored and per-machine, and **only** its `worktree:` block overrides the base. The
worktree `name` / `port` in it are stamped by `worktree create` — do not hand-edit them.

## Keeping the reference current

`references/commands.md` is generated. After changing any command's flags or docs in the
jarrin-claude repo, regenerate it rather than editing it:

```sh
claudjar help --full > claude/skills/claudjar/references/commands.md
```

Add `--include-internal` to document the `api` hook commands as well.
